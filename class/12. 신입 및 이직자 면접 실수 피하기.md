# 신입 및 이직자 면접 실수 피하기

::: tip 핵심 요약
QA 면접에서는 “꼼꼼합니다” 같은 추상적인 답변보다 테스트 케이스, 체크리스트, 우선순위 판단 기준처럼 구조가 보이는 답변이 훨씬 설득력 있다.
:::

## 면접에서 피해야 할 답변

신입이나 이직자가 QA 면접에서 자주 하는 실수는 자신의 장점을 너무 추상적으로 말하는 것이다.

| Don't | 왜 아쉬운가 |
| --- | --- |
| 꼼꼼합니다. | 누구나 말할 수 있고, QA 방식이 보이지 않음 |
| 버그를 잘 찾습니다. | 어떤 기준으로 발견하는지 설명이 없음 |
| 열심히 하겠습니다. | 실무 준비도나 판단 기준이 드러나지 않음 |
| 경계값 분석을 사용했습니다. | 용어만 있고 실제 적용 사례가 없음 |

면접관은 “이 사람이 좋은 사람인가?”만 보는 것이 아니라, 실제 QA 업무에서 구조적으로 사고하고 품질을 판단할 수 있는지를 확인한다.

## 더 나은 답변 방향

좋은 답변은 내가 어떤 결과물을 만들었고, 어떤 기준으로 판단했는지를 보여준다.

| Do | 왜 좋은가 |
| --- | --- |
| 체계적인 테스트 케이스 문서를 작성했습니다. | 기능을 검증 가능한 단위로 나눌 수 있음 |
| 우선순위가 표시된 QA 체크리스트를 만들었습니다. | 제한된 시간 안에서 중요한 것을 먼저 볼 수 있음 |
| 사용자 영향도와 리스크를 기준으로 테스트 범위를 정했습니다. | 실무적인 판단 기준이 보임 |
| 발견한 문제를 재현 절차, 기대 결과, 실제 결과로 정리했습니다. | 협업 가능한 방식으로 이슈를 전달할 수 있음 |

::: info 실무 포인트
면접 성공의 핵심은 “구조를 가진 QA”로 보이는 것이다. 내가 어떤 기준으로 테스트하고 판단하는지 보여주면 실무 준비도가 훨씬 잘 드러난다.
:::

## QA의 역할을 기준으로 답변하기

QA의 핵심 질문은 아래와 같다.

> 이 기능이 과연 문제없이 사용자에게 출시되어도 되는가?

이 질문에 답하려면 단순히 버그를 찾는 것에서 끝나면 안 된다.

| QA가 봐야 할 질문 | 답변에 녹일 내용 |
| --- | --- |
| 가장 큰 리스크는 무엇인가? | 실패했을 때 사용자나 서비스에 미치는 영향 |
| 사용자에게 중요한 기능은 무엇인가? | 회원가입, 로그인, 결제처럼 핵심 흐름 |
| 어떤 기능을 먼저 테스트해야 하는가? | 사용자 영향도, 복구 난이도, 사용 빈도 기준 |
| 발견한 문제를 어떻게 공유했는가? | 재현 절차, 기대 결과, 실제 결과, 환경 정보 |

## 테스트 우선순위 답변 구조

모든 기능을 다 테스트할 수 없기 때문에 우선순위 기준을 설명할 수 있어야 한다.

| 우선순위 | 기준 | 예시 |
| --- | --- | --- |
| 1순위 | 사용자 영향이 큰 기능 | 회원가입, 결제 |
| 2순위 | 실패 시 복구가 어려운 기능 | 결제 승인, 주문 생성 |
| 3순위 | 자주 쓰이는 기능 | 로그인, 검색, 장바구니 |

### 답변 예시

모든 기능을 같은 깊이로 테스트할 수는 없기 때문에 사용자 영향도, 실패 시 리스크, 사용 빈도를 기준으로 우선순위를 정했습니다. 예를 들어 결제 기능은 실패하면 금전 손실이나 주문 데이터 불일치로 이어질 수 있어 1순위로 두고, 로그인처럼 자주 사용되는 기능도 사용자 불편이 크기 때문에 주요 테스트 범위에 포함했습니다.

## 구조화된 답변 연습하기

좋은 결과물을 가지고 있어도 말로 설명하지 못하면 면접에서 충분히 전달되지 않는다.

답변은 아래 순서로 연습하면 좋다.

1. 어떤 기능을 테스트했는지 말한다.
2. 어떤 기준으로 범위와 우선순위를 정했는지 말한다.
3. 정상 흐름과 예외 흐름을 어떻게 나누었는지 말한다.
4. 어떤 문제를 발견했고 어떻게 기록했는지 말한다.
5. 그 과정에서 배운 점을 덧붙인다.

## 포트폴리오로 자신감 높이기

정리한 테스트 케이스와 QA 체크리스트는 포트폴리오의 근거가 될 수 있다.

| 포트폴리오 구성 | 넣을 내용 |
| --- | --- |
| 테스트 대상 | 어떤 기능을 테스트했는지 |
| 테스트 설계 | 정상 흐름, 입력값 유효성, 예외 상황 분류 |
| 체크리스트 | 테스트 전, 중, 후 확인 항목 |
| 이슈 기록 | 발견한 문제의 재현 절차와 기대/실제 결과 |
| 배운 점 | 테스트 관점이나 협업 방식에서 얻은 점 |

포트폴리오는 면접관에게 준비된 인상을 주는 동시에, 스스로 답변할 때도 흔들리지 않게 도와준다.

## QA 면접 치트 시트 만들기

면접 전에 아래 3단 구성으로 나만의 치트 시트를 만들어두면 좋다.

| 항목 | 작성 예시 |
| --- | --- |
| 핵심 결함 사례 | 중복 이메일 입력 시 명확한 오류 메시지가 표시되지 않음 |
| 사용한 QA 기준 | 입력값 유효성 테스트, 오류 메시지 관찰, 사용자 영향도 판단 |
| 결과 및 배운 점 | 재현 절차와 기대/실제 결과를 정리해 공유했고, 사용자 안내의 중요성을 배움 |

이렇게 정리해두면 돌발 질문이 나와도 “기능 -> 기준 -> 사례 -> 배운 점” 흐름으로 답변할 수 있다.

## 면접 질문 예시

> 신입 QA로서 본인의 강점을 어떻게 설명하시겠어요?

## 면접 답변 예시

저는 단순히 꼼꼼하다는 말보다, 구조적으로 테스트하려고 노력한 점을 강점으로 말하고 싶습니다. 회원가입 기능을 기준으로 정상 흐름, 입력값 유효성, 예외 상황으로 테스트 케이스를 나누었고, 사용자 영향도와 실패 시 리스크를 기준으로 우선순위를 정했습니다. 또한 발견한 문제는 재현 절차, 기대 결과, 실제 결과로 정리해 공유할 수 있게 기록했습니다. 이런 방식으로 감이 아니라 기준을 가지고 품질을 판단하는 QA가 되고자 합니다.

## 복습 질문

- “꼼꼼합니다”라는 답변이 QA 면접에서 부족하게 들리는 이유는 무엇일까?
- 구조를 가진 QA처럼 보이려면 어떤 결과물을 근거로 말해야 할까?
- 테스트 우선순위를 설명할 때 어떤 기준을 사용하면 좋을까?
- 나만의 QA 면접 치트 시트에 넣을 결함 사례는 무엇이 있을까?
