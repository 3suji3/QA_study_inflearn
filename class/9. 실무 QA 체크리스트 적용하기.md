# 실무 QA 체크리스트 적용하기

::: tip 핵심 요약
QA 체크리스트는 감으로 테스트하지 않기 위해 테스트 전, 중, 후에 확인할 일을 구조화하는 도구다.
:::

## QA 체크리스트가 필요한 이유

QA는 단순히 버그 하나하나를 쫓는 사람이 아니다. 제품이나 기능이 사용자에게 문제없이 배포되어도 되는지 최종적으로 판단하고, 품질을 책임지는 역할을 한다.

그래서 실무 QA는 개인의 감이나 직관에만 의존하면 안 된다. 어떤 기준으로 준비하고, 무엇을 관찰하고, 어떻게 결과를 공유할지 구조가 있어야 한다.

::: info 실무 포인트
QA의 핵심 철학은 “감이 아니라 구조”다. 체크리스트는 이 구조를 실제 업무에서 실행할 수 있게 만드는 도구다.
:::

## 3단계 테스트 워크플로우

실무 QA 체크리스트는 테스트 전, 테스트 중, 테스트 후로 나누어 관리하면 좋다.

| 단계 | 핵심 목적 | 주요 활동 |
| --- | --- | --- |
| 테스트 전 | 테스트 준비 | 환경 설정, 데이터 준비, 시나리오 검토 |
| 테스트 중 | 기능 검증과 관찰 | 입력값 주입, 상태 모니터링, 오류 기록 |
| 테스트 후 | 결과 판단과 공유 | 결과 분석, 리포트 작성, 팀 공유 |

이 흐름을 기준으로 정리하면 테스트 과정에서 놓치는 부분을 줄일 수 있다.

## 1. 테스트 전 체크리스트

테스트 전에는 “테스트를 제대로 실행할 수 있는 상태인가?”를 확인해야 한다.

| 점검 항목 | 확인할 것 |
| --- | --- |
| 테스트 환경 | 서버, DB, 브라우저, 앱 버전이 테스트 대상과 맞는지 |
| 테스트 계정 | 정상 계정, 신규 계정, 제한 계정 등 필요한 계정이 준비되었는지 |
| 테스트 데이터 | 입력값, 상품, 주문, 쿠폰 등 필요한 데이터가 준비되었는지 |
| 기능 범위 | 이번 테스트에 포함되는 기능과 제외되는 기능이 명확한지 |
| 변경 영향 | 이전 변경 사항이 이번 기능에 영향을 줄 수 있는지 |
| 테스트 시나리오 | 정상 흐름, 예외 흐름, 상태 변화 케이스가 준비되었는지 |

테스트 전 준비가 부족하면 테스트 중에 환경 문제와 기능 문제를 구분하기 어려워진다.

## 2. 테스트 중 체크리스트

테스트 중에는 기능이 동작하는지뿐 아니라 사용자의 흐름과 시스템 상태를 함께 관찰한다.

| 점검 항목 | 확인할 것 |
| --- | --- |
| 입력값 처리 | 필수값 누락, 형식 오류, 중복 입력이 의도대로 처리되는지 |
| 오류 메시지 | 사용자가 무엇을 잘못했는지 명확히 안내하는지 |
| 화면 변화 | 로딩, 버튼 상태, 레이아웃, 화면 이동이 자연스러운지 |
| 상태 변화 | 새로고침, 뒤로가기, 중복 클릭 후에도 안전한지 |
| 데이터 연동 | 화면 간 데이터가 정확히 유지되는지 |
| 예외 처리 | 네트워크 오류, 서버 오류, 세션 만료에 적절히 반응하는지 |

## 회원가입 페이지 적용 예시

회원가입 페이지에서는 아래처럼 체크리스트를 실제 테스트에 적용할 수 있다.

| 상황 | 확인 포인트 | 기대 결과 |
| --- | --- | --- |
| 잘못된 이메일 입력 | 오류 메시지가 명확한지 | 이메일 형식을 확인하라는 안내 표시 |
| 필수값 누락 | 사용자가 무엇을 빠뜨렸는지 알 수 있는지 | 필수 입력 안내 표시 |
| 새로고침 | 입력 데이터가 어떻게 처리되는지 | 일반 정보는 정책에 맞게 유지 또는 초기화 |
| 새로고침 | 민감 정보가 남는지 | 비밀번호 같은 민감 정보는 유지되지 않음 |
| 가입 완료 후 뒤로가기 | 이전 페이지에 민감 정보가 남는지 | 비밀번호 등 민감 정보가 노출되지 않음 |
| 가입 버튼 중복 클릭 | 요청이 여러 번 발생하는지 | 중복 계정 생성 방지 |

특히 회원가입처럼 계정이 생성되는 기능은 상태 변화가 중요하다. 새로고침, 뒤로가기, 중복 클릭에서 사용자 정보가 안전하게 처리되는지 확인해야 한다.

## 3. 테스트 후 체크리스트

테스트 후에는 결과를 정리하고, 이슈를 공유하며, 배포 판단에 필요한 정보를 만든다.

| 점검 항목 | 확인할 것 |
| --- | --- |
| 결과 정리 | Pass, Fail, Blocked 등 실행 결과가 기록되었는지 |
| 이슈 공유 | 발견한 문제를 Jira, Slack 등으로 공유했는지 |
| 증거 첨부 | 스크린샷, 영상, 로그 등 재현 자료가 있는지 |
| 영향도 판단 | 사용자 영향도와 우선순위를 함께 정리했는지 |
| 재테스트 여부 | 수정 후 다시 확인해야 할 항목이 정리되었는지 |
| 남은 리스크 | 테스트하지 못한 범위나 남은 위험을 공유했는지 |

테스트 후 단계는 단순히 “테스트가 끝났다”를 알리는 시간이 아니다. 팀이 배포 여부를 판단할 수 있도록 근거를 정리하는 과정이다.

## 테스트 우선순위 다시 잡기

현실적으로 모든 기능을 같은 깊이로 테스트할 수는 없다. 체크리스트를 적용할 때도 우선순위를 함께 고려해야 한다.

| 우선순위 | 기준 | 예시 |
| --- | --- | --- |
| 1순위 | 사용자 영향도가 큰 핵심 기능 | 결제, 회원가입, 로그인 |
| 2순위 | 장애 발생 시 복구가 어려운 기능 | 주문 생성, 개인정보 변경, 데이터 삭제 |
| 3순위 | 자주 사용되거나 최근 변경된 기능 | 검색, 장바구니, 수정된 API 연동 |

체크리스트는 모든 항목을 기계적으로 확인하기 위한 목록이 아니라, 중요한 항목을 빠뜨리지 않기 위한 기준으로 사용해야 한다.

## 면접에서 활용하는 방법

면접에서 “테스트를 어떻게 설계하고 수행했나요?”라는 질문을 받으면 “꼼꼼하게 봤습니다”라고만 답하면 추상적이다.

대신 아래처럼 구조를 말하면 실무 준비도를 더 잘 보여줄 수 있다.

> 테스트 전에는 환경과 범위를 확인하고, 테스트 중에는 입력값 처리, 상태 변화, 예외 처리를 체크리스트로 관찰했습니다. 테스트 후에는 결과를 정리하고 이슈를 공유한 뒤 재테스트 필요 여부를 판단했습니다.

## 체크리스트 예시 템플릿

| 단계 | 체크 항목 | 결과 | 메모 |
| --- | --- | --- | --- |
| 테스트 전 | 테스트 환경 확인 | Pass / Fail | 서버, DB, 앱 버전 |
| 테스트 전 | 테스트 데이터 준비 | Pass / Fail | 신규 계정, 중복 계정 |
| 테스트 중 | 정상 흐름 확인 | Pass / Fail | 회원가입 성공 |
| 테스트 중 | 오류 메시지 확인 | Pass / Fail | 필수값 누락, 형식 오류 |
| 테스트 중 | 상태 변화 확인 | Pass / Fail | 새로고침, 뒤로가기 |
| 테스트 후 | 이슈 공유 완료 | Pass / Fail | Jira 링크 |
| 테스트 후 | 재테스트 항목 정리 | Pass / Fail | 수정 후 확인 필요 |

## 복습 질문

- 테스트 전 체크리스트에서 환경 확인이 중요한 이유는 무엇일까?
- 회원가입 완료 후 뒤로가기를 눌렀을 때 어떤 정보를 확인해야 할까?
- 테스트 후 결과 정리는 배포 판단에 어떤 도움을 줄까?
- 면접에서 “꼼꼼하다” 대신 어떤 구조로 답변하면 좋을까?
