# 실무 QA의 하루 흐름 파악하기

::: tip 핵심 요약
QA는 단순히 버그를 찾는 사람이 아니라, 기능이 사용자에게 나가도 되는지 판단하는 역할을 한다.
:::

## 전체 흐름

실무 QA의 하루는 보통 아래 순서로 흘러간다.

| 단계 | 하는 일 | 목적 |
| --- | --- | --- |
| 배포 기능 확인 | Jira 같은 프로젝트 관리 도구에서 배포 예정 기능 확인 | 오늘 무엇을 테스트할지 파악 |
| 요구사항 확인 | 기능 명세서, 디자인 문서, 변경 요구사항 확인 | 테스트 기준 정리 |
| 테스트 범위 결정 | 체크리스트 보완, 우선순위 설정 | 중요한 기능부터 검증 |
| 테스트 수행 | 테스트 케이스에 따라 실제 기능 확인 | 기대 결과와 실제 결과 비교 |
| 버그 리포트 작성 | 문제 상황을 재현 가능하게 기록 | 개발자가 빠르게 원인 파악 |
| 재테스트 | 수정된 기능을 다시 확인 | 이슈 해결 여부 검증 |
| 배포 판단 | 기능이 문제없이 나가도 되는지 판단 | 최종 품질 기준 확인 |

## 1. 배포될 기능 확인

QA는 먼저 오늘 배포될 기능이 무엇인지 확인한다.

- 어떤 기능이 배포 예정인지 확인한다.
- 기능 명세서와 디자인 문서를 확인한다.
- 요구사항이 변경된 부분이 있는지 살핀다.
- 테스트해야 할 범위와 우선순위를 정한다.

이 단계가 중요한 이유는 QA가 모든 기능을 같은 깊이로 볼 수 없기 때문이다. 사용자 영향이 큰 기능, 변경이 많은 기능, 장애 위험이 큰 기능부터 우선적으로 확인해야 한다.

## 2. 테스트 수행

테스트는 기대 결과와 실제 결과를 비교하는 과정이다.

| 항목 | 의미 |
| --- | --- |
| 테스트 절차 | 어떤 순서로 기능을 확인할지 |
| 기대 결과 | 정상이라면 어떤 결과가 나와야 하는지 |
| 실제 결과 | 실제 테스트에서 어떤 결과가 나왔는지 |
| 판정 | Pass 또는 Fail |

예상과 다르게 오류 메시지가 뜨거나 기능이 정상 동작하지 않으면, 실제 결과를 자세히 기록하고 `Fail`로 표시한다.

## 3. 좋은 버그 리포트 작성

좋은 버그 리포트는 개발자가 문제를 빠르고 정확하게 이해할 수 있게 만든다.

| 항목 | 작성 내용 |
| --- | --- |
| 제목 | 문제의 핵심이 한눈에 보이게 작성 |
| 재현 경로 | 문제를 다시 발생시킬 수 있는 단계 |
| 기대 결과 | 정상적으로 동작했을 때의 상태 |
| 실제 결과 | 현재 발생한 오류나 이상 동작 |
| 환경 정보 | OS, 기기, 앱 버전, 브라우저 등 |

버그를 기록할 때는 심각도와 우선순위도 함께 생각하면 좋다.

| 구분 | 의미 | 예시 |
| --- | --- | --- |
| 심각도 | 문제가 서비스에 주는 영향 | 결제가 불가능함, 앱이 종료됨 |
| 우선순위 | 얼마나 빨리 수정해야 하는지 | 오늘 배포 전 반드시 수정 |

심각도가 높다고 항상 우선순위가 가장 높은 것은 아니다. 예를 들어 특정 구형 기기에서만 드물게 발생하는 문제와, 많은 사용자가 매일 겪는 로그인 오류는 수정 순서가 다를 수 있다.

::: info 실무 포인트
버그 리포트의 목적은 “문제가 있다”를 알리는 것이 아니라, 개발자가 같은 문제를 빠르게 재현하고 수정할 수 있게 돕는 것이다.
:::

## 4. 버그 리포트 이후 흐름

버그를 등록한 뒤에도 QA의 일이 끝나는 것은 아니다.

1. QA가 이슈 리포트를 등록한다.
2. Jira 또는 Slack으로 이슈를 공유한다.
3. 개발팀이 원인을 분석하고 코드를 수정한다.
4. QA가 수정 사항을 재테스트한다.
5. 문제가 해결되면 이슈를 종결한다.

재테스트할 때는 수정된 부분만 보는 데서 끝나지 않고, 주변 기능이 함께 깨지지 않았는지도 확인해야 한다. 이를 회귀 테스트라고 한다.

| 재테스트 관점 | 확인할 것 |
| --- | --- |
| 수정 확인 | 보고한 문제가 실제로 해결되었는지 |
| 영향 범위 확인 | 수정된 코드와 연결된 주변 기능이 정상인지 |
| 재발 방지 | 같은 조건에서 문제가 다시 발생하지 않는지 |

## 최종 판단

QA의 마지막 질문은 아래와 같다.

> 이 기능이 문제없이 사용자에게 나가도 되는가?

이 질문에 확신을 가지고 답할 수 있을 때 배포 승인을 내릴 수 있다.

## 면접 질문 예시

> 실무 QA는 하루 동안 어떤 흐름으로 업무를 진행하나요?

## 면접 답변 예시

실무 QA는 먼저 배포 예정 기능과 요구사항을 확인하고, 테스트 범위와 우선순위를 정합니다. 이후 테스트 케이스를 기준으로 기대 결과와 실제 결과를 비교하며 검증하고, 발견한 이슈는 재현 경로와 환경 정보를 포함해 리포트합니다. 수정 후에는 재테스트와 회귀 테스트를 통해 문제가 해결되었는지 확인한 뒤, 최종적으로 배포 가능 여부를 판단합니다.

## 복습 질문

- QA가 테스트 전 요구사항을 먼저 확인해야 하는 이유는 무엇일까?
- 좋은 버그 리포트와 불친절한 버그 리포트의 차이는 무엇일까?
- 심각도와 우선순위는 어떤 기준으로 다르게 판단해야 할까?
- 수정 완료된 버그를 재테스트할 때 어떤 기준으로 확인해야 할까?
