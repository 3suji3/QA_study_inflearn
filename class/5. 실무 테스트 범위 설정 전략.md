# 실무 테스트 범위 설정 전략

::: tip 핵심 요약
실무 QA에서는 모든 것을 100% 테스트할 수 없기 때문에, 사용자 영향도와 위험도를 기준으로 중요한 것부터 테스트 범위를 정한다.
:::

## 모든 것을 테스트할 수 없는 이유

실무에서는 테스트할 기능보다 사용할 수 있는 자원이 항상 부족하다.

- 테스트 시간이 제한되어 있다.
- 예산과 인력이 한정되어 있다.
- 기능과 예외 케이스가 너무 많다.
- 배포 일정 안에서 품질 판단을 내려야 한다.

그래서 QA는 “전부 다 확인하겠다”가 아니라, “무엇을 먼저 깊게 볼 것인가”를 결정해야 한다.

## 테스트 우선순위 기준

테스트 범위는 중요도와 위험도를 기준으로 정한다.

| 기준 | 설명 | 예시 |
| --- | --- | --- |
| 사용자 영향이 큰 기능 | 문제가 생기면 많은 사용자가 불편을 겪는 기능 | 회원가입, 로그인, 결제 |
| 복구가 어려운 기능 | 실패했을 때 데이터 손상이나 금전 손실이 생길 수 있는 기능 | 결제 승인, 주문 생성, 개인정보 변경 |
| 자주 쓰이는 기능 | 사용 빈도가 높아 작은 오류도 크게 느껴지는 기능 | 검색, 게시물 작성, 장바구니 |
| 최근 변경된 기능 | 기존에는 되던 기능도 변경 이후 오류가 생길 수 있음 | 리팩터링된 로그인, 수정된 결제 API |
| 외부 연동 기능 | 우리 시스템 밖의 변수까지 고려해야 하는 기능 | PG사 결제, 문자 발송, 소셜 로그인 |

::: info 실무 포인트
테스트 범위 설정은 “덜 중요한 것은 안 본다”가 아니라, 제한된 시간 안에서 가장 위험한 부분을 먼저 확인하는 전략이다.
:::

## 이커머스 결제 흐름 예시

온라인 쇼핑 서비스의 결제 흐름을 기준으로 테스트 우선순위를 나누면 아래처럼 볼 수 있다.

| 단계 | 사용자 영향도 | 복구 난이도 | 우선순위 |
| --- | --- | --- | --- |
| 장바구니 | 구매 의사 결정 단계 | 낮음 | 보통 |
| 주문서 작성 | 배송지, 연락처 등 정보 입력 | 낮음 | 보통 |
| 결제 수단 선택 | 결제 방식 결정 | 보통 | 높음 |
| 결제 진행 | 금전 거래가 발생하는 핵심 단계 | 높음 | 매우 높음 |
| 주문 완료 | 결제 결과와 주문 데이터가 확정되는 단계 | 높음 | 매우 높음 |

결제 진행과 주문 완료는 가장 높은 우선순위로 테스트해야 한다. 결제는 실패했을 때 금전 손실, 주문 데이터 불일치, 고객 문의 증가로 이어질 수 있기 때문이다.

## 위험도 분석으로 범위 확장하기

기본 핵심 기능 외에도 아래 항목은 테스트 범위에 포함할지 우선 검토해야 한다.

| 항목 | 테스트가 필요한 이유 |
| --- | --- |
| 신규 추가 기능 | 아직 충분히 검증되지 않았기 때문 |
| 복잡한 비즈니스 로직 | 조건이 많아 예외 케이스가 발생하기 쉽기 때문 |
| 외부 서비스 연동 | 외부 시스템 상태에 따라 실패할 수 있기 때문 |
| 최근 변경된 코드 | 주변 기능에 예상치 못한 영향이 생길 수 있기 때문 |

예를 들어 결제 기능이 직접 변경되지 않았더라도, 주문서 작성 로직이나 쿠폰 적용 로직이 바뀌었다면 결제 결과에 영향을 줄 수 있다. 이 경우 관련 기능까지 회귀 테스트 범위에 넣어야 한다.

## 테스트 범위 정의서에 들어갈 내용

실무에서는 QA 혼자 테스트 범위를 정하지 않는다. 개발자, 기획자 등 관련자가 함께 범위를 협의하고 문서로 남긴다.

| 항목 | 작성 내용 |
| --- | --- |
| 테스트 대상 | 어떤 기능을 테스트할지 |
| 제외 범위 | 이번 테스트에서 제외하는 기능은 무엇인지 |
| 우선순위 | 어떤 항목을 먼저 깊게 볼지 |
| 위험 요소 | 실패했을 때 영향이 큰 부분은 무엇인지 |
| 테스트 환경 | OS, 브라우저, 기기, 서버 환경 |
| 완료 기준 | 어떤 조건이면 테스트를 종료할 수 있는지 |

테스트 범위 정의서는 나중에 “왜 이 기능은 테스트했고, 왜 이 기능은 제외했는지”를 설명하는 기준이 된다.

## 제외 범위 기록 원칙

테스트하지 않는 범위를 정할 때도 이유를 남겨야 한다. 제외 범위를 기록하지 않으면 나중에 문제가 생겼을 때 “왜 확인하지 않았는지”를 설명하기 어렵다.

| 제외 기준 | 기록 예시 |
| --- | --- |
| 변경 영향이 낮음 | 이번 배포에서 코드 변경이 없고 기존 회귀 테스트 결과가 안정적임 |
| 사용 빈도가 낮음 | 내부 관리자만 사용하는 기능이며 이번 배포 핵심 흐름과 무관함 |
| 일정상 제외 | 배포 일정상 이번 사이클에서는 스모크 테스트만 수행 |
| 별도 검증 예정 | 외부 연동 테스트는 스테이징 환경 준비 후 별도 진행 |

제외한 기능이 있다고 해서 QA 책임이 줄어드는 것은 아니다. 중요한 것은 제외 이유와 남은 위험을 팀이 함께 이해하고 결정하는 것이다.

## 범위 설정 체크리스트

테스트 범위를 정할 때 아래 질문을 순서대로 확인한다.

1. 이 기능이 실패하면 사용자에게 얼마나 큰 영향을 주는가?
2. 문제가 생겼을 때 복구가 쉬운가, 어려운가?
3. 사용 빈도가 높은 기능인가?
4. 최근에 변경된 기능인가?
5. 외부 서비스나 다른 기능과 연결되어 있는가?
6. 이번 배포에서 반드시 확인해야 하는 완료 기준은 무엇인가?

## 면접 질문 예시

> 시간이 부족한 상황에서 테스트 범위와 우선순위는 어떻게 정하시나요?

## 면접 답변 예시

실무에서는 모든 기능을 같은 깊이로 테스트할 수 없기 때문에 사용자 영향도와 위험도를 기준으로 우선순위를 정합니다. 예를 들어 이커머스에서는 장바구니보다 결제 진행과 주문 완료의 우선순위를 높게 두고, 금전 손실이나 데이터 불일치가 발생하지 않는지 집중적으로 확인합니다.

## 복습 질문

- 테스트 범위에서 제외한 기능은 어떻게 기록해야 할까?
- 결제 기능에서 “성공”은 어느 시점까지 확인해야 할까?
- 최근 변경된 코드가 직접 수정된 기능이 아니어도 테스트 범위에 들어갈 수 있는 이유는 무엇일까?
- 테스트하지 않기로 한 범위의 남은 위험은 누구와 공유해야 할까?
