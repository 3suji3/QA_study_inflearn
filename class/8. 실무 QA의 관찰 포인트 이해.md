# 실무 QA의 관찰 포인트 이해

::: tip 핵심 요약
실무 QA는 버튼을 눌러 기능이 동작하는지만 보지 않고, 화면 변화, 오류 메시지, 데이터 흐름, 상태 변화까지 함께 관찰해야 한다.
:::

## 관찰 포인트가 중요한 이유

QA의 관찰은 단순한 기능 확인에서 끝나지 않는다.

기능이 “동작한다”는 사실만으로는 충분하지 않다. 사용자가 기능을 쓰는 동안 화면이 자연스럽게 변하는지, 데이터가 정확히 이어지는지, 예외 상황에서도 시스템이 안전하게 반응하는지까지 봐야 한다.

| 단순 확인 | QA 관찰 |
| --- | --- |
| 버튼을 클릭했더니 동작한다 | 클릭 후 로딩, 화면 변화, 데이터 반영, 오류 가능성까지 본다 |
| 에러 메시지가 뜬다 | 메시지가 명확하고 사용자가 다음 행동을 알 수 있는지 본다 |
| 장바구니에 상품이 담긴다 | 상품명, 수량, 가격, 옵션이 다음 화면까지 정확히 이어지는지 본다 |

::: info 실무 포인트
기능은 동작하지만 사용자가 헷갈리거나 데이터가 어긋난다면, QA 관점에서는 충분히 이슈가 될 수 있다.
:::

## 1. 화면 변화 관찰

사용자가 어떤 동작을 했을 때 화면이 어떻게 변하는지 면밀히 봐야 한다.

| 관찰 항목 | 확인할 것 |
| --- | --- |
| 로딩 상태 | 버튼 클릭 후 로딩 스피너나 진행 상태가 표시되는지 |
| 레이아웃 | 데이터 로딩 중 화면이 깨지거나 텍스트가 겹치지 않는지 |
| 버튼 상태 | 요청 중 버튼이 중복 클릭되지 않도록 비활성화되는지 |
| 화면 이동 | 성공 또는 실패 후 의도한 화면으로 이동하는지 |
| 반응성 | 모바일, 태블릿, 데스크톱에서 UI가 자연스럽게 보이는지 |

예를 들어 기능은 정상 처리되었더라도 로딩 중 텍스트가 겹치거나 버튼을 여러 번 누를 수 있다면 사용자는 불편을 겪을 수 있다.

## 2. 오류 메시지 관찰

오류 메시지는 단순히 “오류가 발생했습니다”라고 알려주는 데서 끝나면 부족하다.

좋은 오류 메시지는 사용자가 무엇을 잘못했는지, 다음에 무엇을 해야 하는지 알 수 있게 도와야 한다.

| 나쁜 메시지 | 더 나은 메시지 |
| --- | --- |
| 오류가 발생했습니다. | 이메일 형식이 올바르지 않습니다. |
| 실패했습니다. | 네트워크 연결을 확인한 뒤 다시 시도해주세요. |
| 입력 오류 | 비밀번호는 8자 이상이어야 합니다. |

오류 메시지를 볼 때는 아래 기준으로 확인한다.

- 원인이 명확한가?
- 사용자가 다음 행동을 알 수 있는가?
- 너무 기술적인 표현을 그대로 노출하지 않는가?
- 같은 상황에서 일관된 메시지가 표시되는가?

## 3. 데이터 연동 관찰

QA는 화면에 보이는 결과뿐 아니라 데이터가 흐름에 따라 정확히 이어지는지도 확인해야 한다.

이커머스의 장바구니 흐름을 예로 들면 아래처럼 볼 수 있다.

| 흐름 | 관찰할 데이터 |
| --- | --- |
| 상품 상세 | 상품명, 가격, 옵션, 수량 |
| 장바구니 | 담은 상품 정보가 정확히 반영되는지 |
| 주문서 | 장바구니의 상품, 수량, 가격이 유지되는지 |
| 결제 | 최종 결제 금액과 주문 정보가 일치하는지 |
| 주문 완료 | 결제 결과와 주문 내역이 정확히 생성되는지 |

장바구니에서는 수량이 2개였는데 결제 페이지에서 1개로 표시된다면 데이터 불일치다. 이런 문제는 사용자 불편을 넘어서 결제 오류나 주문 데이터 문제로 이어질 수 있다.

## 4. 상태 변화 관찰

사용자는 항상 정해진 순서대로만 움직이지 않는다. 새로고침, 뒤로가기, 중복 클릭 같은 행동도 테스트해야 한다.

| 사용자 행동 | 발생할 수 있는 문제 | 확인할 것 |
| --- | --- | --- |
| 새로고침 | 입력 폼 데이터 초기화 | 저장되어야 할 값과 초기화되어도 되는 값을 구분 |
| 새로고침 | 결제 중복 요청 | 같은 요청이 두 번 처리되지 않는지 |
| 뒤로가기 | 이전 상태 유지 실패 | 이전 화면의 데이터가 의도대로 남아 있는지 |
| 뒤로가기 | 캐시로 인한 데이터 불일치 | 최신 데이터가 필요한 화면에서 오래된 정보가 보이지 않는지 |
| 세션 만료 | 작업 도중 로그아웃 처리 | 안내 메시지와 재로그인 흐름이 제공되는지 |

특히 결제, 주문, 회원가입처럼 데이터가 생성되는 기능에서는 새로고침과 중복 클릭이 큰 이슈가 될 수 있다.

## 실무 QA 체크리스트

체크리스트는 테스트 전, 중, 후 세 단계로 나누어 관리하면 좋다.

| 단계 | 핵심 점검 항목 |
| --- | --- |
| 테스트 전 | 서버, DB, 계정, 테스트 데이터, 기능 범위 확인 |
| 테스트 전 | 이전 변경 사항이 이번 기능에 미치는 영향 파악 |
| 테스트 중 | 입력값 처리, 화면 변화, 상태 변화 관찰 |
| 테스트 중 | 예외 상황과 오류 메시지가 의도대로 동작하는지 확인 |
| 테스트 중 | 화면 간 데이터 연동이 정확한지 확인 |
| 테스트 후 | 테스트 결과 정리, 이슈 공유, 재테스트 여부 판단 |

## 관찰 결과를 기록하는 방법

관찰한 내용을 기록할 때는 “이상해 보인다”보다 구체적인 증거를 남기는 것이 좋다.

| 기록 항목 | 예시 |
| --- | --- |
| 발생 위치 | 장바구니에서 주문서로 이동하는 시점 |
| 사용자 행동 | 수량 2개 선택 후 주문서 작성 클릭 |
| 기대 결과 | 주문서에도 수량 2개가 표시되어야 함 |
| 실제 결과 | 주문서에서 수량 1개로 표시됨 |
| 영향 | 결제 금액이 잘못 계산될 수 있음 |

이렇게 기록하면 개발자나 기획자가 문제를 더 빠르게 이해할 수 있다.

## 면접 답변 예시

테스트를 수행할 때는 기능이 동작하는지만 보지 않고 화면 변화, 오류 메시지, 데이터 연동, 새로고침과 뒤로가기 같은 상태 변화까지 관찰합니다. 예를 들어 장바구니에서 주문서로 이동할 때 상품명, 수량, 가격, 옵션이 정확히 유지되는지 확인하고, 불일치가 있으면 사용자 영향과 함께 이슈로 기록합니다.

## 복습 질문

- 기능이 정상 동작해도 QA 이슈가 될 수 있는 화면 문제에는 무엇이 있을까?
- 좋은 오류 메시지는 어떤 조건을 만족해야 할까?
- 장바구니에서 결제까지 데이터 연동을 확인할 때 어떤 값을 비교해야 할까?
- 새로고침과 뒤로가기는 왜 실무 QA에서 중요한 관찰 포인트일까?
