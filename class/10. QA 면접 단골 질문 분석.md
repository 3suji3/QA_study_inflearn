# QA 면접 단골 질문 분석

::: tip 핵심 요약
QA 면접 질문은 지식 암기 여부보다 논리적으로 사고하고, 문제를 구조화하며, 실무 관점으로 설명할 수 있는지를 확인하기 위한 질문이다.
:::

## 면접 질문의 진짜 의도

면접관은 단순히 “테스트 기법을 아는가?”만 확인하지 않는다.

| 면접관이 보고 싶은 것 | 설명 |
| --- | --- |
| 논리적 사고 | 기능을 어떤 기준으로 나누고 판단하는지 |
| 문제 해결 과정 | 문제가 생겼을 때 어떻게 원인을 좁혀 가는지 |
| 실무 이해도 | 실제 QA 업무 흐름을 알고 있는지 |
| 구조화 능력 | 추상적인 생각을 테스트 케이스나 체크리스트로 바꿀 수 있는지 |
| 설명력 | 자신의 판단 기준을 다른 사람에게 명확히 전달할 수 있는지 |

“저는 꼼꼼합니다”, “최선을 다하겠습니다” 같은 답변은 면접관의 궁금증을 해결해주기 어렵다. 면접에서는 내가 어떤 기준으로 테스트를 설계하고 판단하는지를 구체적으로 보여줘야 한다.

## 질문 1. 테스트는 어떻게 설계했나요?

이 질문은 단순히 테스트 케이스를 써본 적이 있는지 묻는 질문이 아니다. 기능을 어떤 기준으로 쪼개고, 어떤 순서로 검증하는지 확인하는 질문이다.

### 답변 구조

| 단계 | 설명 | 예시 |
| --- | --- | --- |
| 1단계 | 기능을 구조화한다 | 회원가입 기능을 정상 흐름과 예외 흐름으로 나눔 |
| 2단계 | 케이스를 상세화한다 | 입력값, 상태 변화, 보안, UI/UX, 성능 관점 추가 |
| 3단계 | 우선순위를 설명한다 | 사용자 영향이 큰 흐름부터 검증 |

### 답변 공식

```text
저는 [기능] 테스트 시, [우선순위 기준]을 기준으로 [정상/예외] 케이스로 나누어 설계했습니다.
```

### 답변 예시

회원가입 기능을 테스트할 때 사용자 영향이 큰 정상 가입 흐름을 먼저 확인하고, 이후 입력값 유효성, 중복 이메일, 비밀번호 규칙 오류 같은 예외 케이스로 나누어 테스트 케이스를 설계했습니다. 또한 새로고침이나 뒤로가기 같은 상태 변화도 함께 확인해 사용자가 실수했을 때 데이터가 안전하게 처리되는지 보았습니다.

## 질문 2. 테스트 범위와 우선순위는 어떻게 정했나요?

이 질문은 “모든 것을 다 테스트할 수 없다”는 현실을 이해하고 있는지 확인하는 질문이다.

### 우선순위 기준

| 기준 | 설명 | 예시 |
| --- | --- | --- |
| 사용자 영향도 | 문제가 생겼을 때 사용자가 크게 불편한가 | 결제, 회원가입, 로그인 |
| 실패 시 리스크 | 복구가 어렵거나 손실이 큰가 | 결제 오류, 개인정보 변경, 데이터 삭제 |
| 사용 빈도 | 많은 사용자가 자주 쓰는가 | 검색, 장바구니, 게시글 작성 |
| 최근 변경 여부 | 이번 배포에서 수정되었는가 | 수정된 API, 리팩터링된 기능 |

### 답변 예시

테스트 범위는 사용자 영향도, 실패 시 리스크, 사용 빈도를 기준으로 정했습니다. 예를 들어 결제 기능은 실패하면 금전 손실이나 주문 데이터 불일치로 이어질 수 있기 때문에 높은 우선순위로 두고, 장바구니나 주문서와 연결된 데이터 흐름까지 함께 확인했습니다.

## 질문 3. 테스트할 때 체크리스트를 어떻게 활용했나요?

이 질문은 테스트를 감으로 하는지, 체계적인 기준으로 하는지 확인하기 위한 질문이다.

| 단계 | 체크리스트 기준 |
| --- | --- |
| 테스트 전 | 테스트 환경, 기능 범위, 이전 변경 영향 확인 |
| 테스트 중 | 입력값 처리, 상태 변화, 예외 처리 관찰 |
| 테스트 후 | 결과 정리, 이슈 공유, 재테스트 여부 판단 |

### 답변 예시

테스트 전에는 환경과 기능 범위를 확인하고, 테스트 중에는 입력값 처리, 상태 변화, 예외 처리를 체크리스트로 확인했습니다. 테스트 후에는 Pass/Fail 결과와 발견한 이슈를 정리하고, 수정이 필요한 항목은 재테스트 대상으로 분류했습니다. 이렇게 체크리스트를 사용하면 감이 아니라 구조적인 기준으로 테스트할 수 있습니다.

## Bad vs Good 답변 비교

같은 질문이라도 답변 방식에 따라 실무 이해도가 다르게 보인다.

| 구분 | 답변 예시 | 아쉬운 점 또는 장점 |
| --- | --- | --- |
| Bad | 꼼꼼하게 테스트했습니다. | 어떤 기준으로 무엇을 봤는지 알 수 없음 |
| Bad | 경계값 분석을 사용했습니다. | 용어만 말하고 적용 과정이 없음 |
| Good | 회원가입 기능에서 사용자 영향이 큰 정상 흐름부터 확인하고, 비밀번호 규칙 오류 같은 예외 케이스로 나누어 설계했습니다. | 기능, 기준, 케이스 분리가 드러남 |
| Good | 결제 기능은 실패 시 금전 손실이 발생할 수 있어 우선순위를 높게 두고, 주문 생성과 결제 금액 일치 여부를 함께 확인했습니다. | 위험도와 사용자 영향 기준이 보임 |

## 답변을 구조화하는 방법

면접 답변은 아래 순서로 말하면 정리되어 보인다.

1. 어떤 기능을 기준으로 말할지 정한다.
2. 테스트 기준을 먼저 말한다.
3. 정상 흐름과 예외 흐름으로 나눈다.
4. 실제 확인한 포인트를 예시로 든다.
5. 마지막에 QA 관점의 판단 기준을 덧붙인다.

## 실무 경험이 부족할 때 답변하는 법

실무 경험이 많지 않더라도 “실제로 이렇게 생각하고 설계할 수 있다”는 것을 보여주면 된다.

| 피해야 할 답변 | 더 나은 방향 |
| --- | --- |
| 해본 적은 없지만 열심히 하겠습니다. | 실습한 기능을 기준으로 테스트 설계 과정을 설명 |
| 꼼꼼히 보겠습니다. | 어떤 체크리스트로 확인할지 설명 |
| 버그를 많이 찾겠습니다. | 사용자 영향도와 위험도 기준으로 우선순위 설명 |

::: info 실무 포인트
면접에서는 완벽한 경험보다 구조화된 사고가 더 중요하게 보일 때가 많다. 기능을 어떻게 나누고, 어떤 기준으로 판단했는지를 말하면 실무 준비도를 보여줄 수 있다.
:::

## 면접 질문 예시

> QA 면접에서 “테스트를 어떻게 설계했나요?”라는 질문을 받으면 어떻게 답변하시겠어요?

## 면접 답변 예시

회원가입 기능을 예로 들면, 먼저 정상 가입 흐름을 기준으로 사용자가 목적을 달성할 수 있는지 확인했습니다. 이후 입력값 유효성, 중복 이메일, 비밀번호 규칙 오류, 네트워크 오류 같은 예외 케이스로 나누어 테스트 케이스를 설계했습니다. 테스트 범위는 사용자 영향도와 실패 시 리스크를 기준으로 우선순위를 정했고, 테스트 전/중/후 체크리스트를 활용해 환경 확인, 상태 변화 관찰, 결과 정리까지 구조적으로 진행했습니다.

## 복습 질문

- 면접관이 “테스트를 어떻게 설계했나요?”라고 물을 때 진짜 확인하려는 것은 무엇일까?
- “꼼꼼하게 테스트했습니다”라는 답변이 부족한 이유는 무엇일까?
- 테스트 범위와 우선순위를 설명할 때 어떤 기준을 말하면 좋을까?
- 실무 경험이 부족할 때도 구조적으로 답변하려면 어떤 기능 예시를 준비하면 좋을까?
