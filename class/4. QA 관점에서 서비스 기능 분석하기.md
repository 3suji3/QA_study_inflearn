# QA 관점에서 서비스 기능 분석하기

::: tip 핵심 요약
QA 관점의 기능 분석은 “기능이 된다”에서 끝나지 않고, 어떤 조건에서도 안정적으로 동작하는지 확인하는 과정이다.
:::

## 세 가지 관점

같은 기능을 보더라도 역할에 따라 바라보는 기준이 다르다.

| 관점 | 핵심 질문 |
| --- | --- |
| 개발자 관점 | 의도한 대로 구현되었는가? |
| 사용자 관점 | 내가 원하는 목표를 달성할 수 있는가? |
| QA 관점 | 다양한 조건과 예외 상황에서도 안정적인가? |

QA는 기능 구현 여부만 확인하지 않는다. 오류, 예외 상황, 사용자 실수까지 포함해 서비스가 견고하게 동작하는지 탐색한다.

## 기능 분석의 기본 질문

QA가 기능을 분석할 때는 아래 질문을 먼저 던진다.

| 질문 | 확인하는 것 |
| --- | --- |
| 이 기능은 언제 성공인가? | 기대 결과와 완료 기준 |
| 실패는 어떤 경우인가? | 오류, 예외, 장애 상황 |
| 사용자가 실수하면 어떻게 되는가? | 잘못된 입력이나 예상 밖 행동에 대한 대응 |

이 질문들은 테스트 케이스를 만들기 위한 출발점이 된다.

## 성공 조건 정의하기

회원가입 기능의 성공은 단순히 `가입하기` 버튼을 눌렀을 때 오류가 없다는 뜻이 아니다.

| 성공 조건 | 확인 포인트 |
| --- | --- |
| 유효성 검증 통과 | 이메일, 비밀번호, 필수값이 올바른지 |
| 계정 생성 | 데이터베이스에 사용자 정보가 정상 저장되는지 |
| 성공 알림 | 사용자에게 가입 완료 메시지가 표시되는지 |
| 페이지 이동 | 로그인 페이지 또는 다음 단계로 이동하는지 |
| 로그인 가능 여부 | 새 계정 정보로 실제 로그인이 가능한지 |

::: info 실무 포인트
QA에게 성공은 “화면에서 에러가 안 난 상태”가 아니라, 사용자의 목적과 시스템 처리가 모두 완료된 상태다.
:::

## 실패 시나리오 예측하기

기능 분석에서는 실패할 수 있는 조건을 미리 나누어 생각해야 한다.

| 실패 유형 | 예시 | 기대 동작 |
| --- | --- | --- |
| 유효하지 않은 입력 | 이메일 형식 오류, 필수값 누락 | 입력값 오류 메시지 표시 |
| 시스템 오류 | 서버 장애, 데이터베이스 연결 실패 | 사용자에게 실패 안내, 데이터 불일치 방지 |
| 네트워크 문제 | 인터넷 연결 불안정, 요청 타임아웃 | 재시도 안내 또는 안전한 실패 처리 |
| 사용자 오용 | 의도치 않은 반복 클릭, 잘못된 접근 | 중복 처리 방지, 적절한 제한 |

실패 시나리오를 예측하는 이유는 문제를 찾기 위해서만이 아니다. 각 상황에서 사용자가 어떤 피드백을 받아야 하고, 시스템이 어떤 상태를 유지해야 하는지 정의하기 위해서다.

## 질문을 테스트 케이스로 바꾸기

QA의 질문은 구체적인 테스트 케이스로 전환되어야 한다.

| QA 질문 | 테스트 절차 | 기대 결과 |
| --- | --- | --- |
| 이미 가입된 이메일을 사용하면 어떻게 될까? | 가입된 이메일로 회원가입 시도 | `이미 사용 중인 이메일입니다.` 메시지 표시 |
| 필수값을 비워두면 어떻게 될까? | 이메일 또는 비밀번호를 입력하지 않고 가입 시도 | 필수 입력 안내 표시, 가입 처리 차단 |
| 가입 중 네트워크가 끊기면 어떻게 될까? | 가입 요청 중 네트워크 차단 | 중복 계정 생성 없이 실패 안내 |
| 가입 버튼을 여러 번 누르면 어떻게 될까? | 가입하기 버튼 연속 클릭 | 중복 요청 또는 중복 계정 생성 방지 |

이처럼 추상적인 질문을 `절차`와 `기대 결과`로 바꾸는 것이 테스트 설계의 핵심이다.

## 기능 분석 체크리스트

새 기능을 볼 때 아래 순서로 정리하면 테스트 케이스로 옮기기 쉽다.

1. 사용자의 목표를 정의한다.
2. 기능의 성공 조건을 나눈다.
3. 실패할 수 있는 조건을 예측한다.
4. 사용자 실수 상황을 추가한다.
5. 각 상황의 기대 결과를 적는다.
6. 테스트 절차로 바꾼다.

## 분석 관점 확장하기

기능을 더 깊게 분석하려면 화면에서 보이는 동작뿐 아니라 데이터, 권한, 상태, 연동까지 함께 본다.

| 관점 | 질문 |
| --- | --- |
| 데이터 | 저장, 수정, 삭제된 값이 정확하게 반영되는가? |
| 권한 | 접근할 수 없는 사용자가 기능을 사용할 수 없는가? |
| 상태 | 새로고침, 뒤로 가기, 중복 클릭 후에도 흐름이 안전한가? |
| 연동 | 외부 API나 다른 기능이 실패했을 때도 적절히 처리되는가? |
| 안내 | 사용자가 성공, 실패, 다음 행동을 이해할 수 있는가? |

이 관점은 테스트 케이스를 빠뜨리지 않기 위한 보조 질문으로 사용할 수 있다.

## 면접 답변 예시

기능을 분석할 때 먼저 성공 기준을 정의하고, 입력 오류, 시스템 오류, 네트워크 문제, 사용자 실수 같은 실패 시나리오를 나누어 봅니다. 이후 각 질문을 테스트 절차와 기대 결과로 바꾸어 테스트 케이스를 작성합니다.

## 복습 질문

- QA 관점에서 성공 조건을 자세히 정의해야 하는 이유는 무엇일까?
- 네트워크 문제와 시스템 오류는 테스트 케이스에서 어떻게 다르게 다룰 수 있을까?
- 이미 가입된 이메일 테스트에서 확인해야 할 것은 오류 메시지뿐일까?
- 기능을 분석할 때 권한과 상태 관점을 함께 봐야 하는 이유는 무엇일까?
