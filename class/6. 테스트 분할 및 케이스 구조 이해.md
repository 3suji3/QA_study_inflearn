# 테스트 분할 및 케이스 구조 이해

::: tip 핵심 요약
테스트 분할은 기능을 정상 흐름, 입력값 유효성, 예외 상황으로 나누어 테스트의 우선순위와 사고 순서를 정리하는 과정이다.
:::

## 테스트를 나누어야 하는 이유

현실적으로 모든 기능과 모든 조건을 완벽하게 테스트하기는 어렵다. 시간, 인력, 일정이 제한되어 있기 때문이다.

그래서 QA는 중요한 부분을 먼저 보기 위해 테스트 대상을 명확히 나누어야 한다.

| 분할 기준 | 확인하는 질문 |
| --- | --- |
| 정상 흐름 | 이 기능은 언제 성공인가? |
| 입력값 유효성 | 어떤 값은 허용되고, 어떤 값은 막아야 하는가? |
| 예외 상황 | 실패하거나 사용자가 실수하면 어떻게 되는가? |

테스트 분할은 단순히 기능을 쪼개는 작업이 아니라, QA의 시선으로 테스트 우선순위와 사고의 순서를 만드는 첫 단계다.

## 1. 정상 흐름 테스트

정상 흐름은 사용자가 아무 문제 없이 기능의 목적을 달성하는 이상적인 경로다. 흔히 해피 패스라고도 부른다.

회원가입 기능의 정상 흐름은 아래처럼 볼 수 있다.

| 순서 | 사용자 행동 | 기대 결과 |
| --- | --- | --- |
| 1 | 이메일 입력 | 유효한 이메일로 인식 |
| 2 | 비밀번호 설정 | 비밀번호 규칙 통과 |
| 3 | 약관 동의 | 필수 약관 동의 처리 |
| 4 | 가입하기 클릭 | 계정 생성 요청 |
| 5 | 가입 완료 | 성공 메시지 표시 또는 로그인 페이지 이동 |

정상 흐름 테스트는 가장 먼저 확인해야 한다. 기본 성공 경로가 깨져 있으면 예외 케이스를 확인하기 전에 핵심 기능 자체가 동작하지 않는 상태이기 때문이다.

## 2. 입력값 유효성 테스트

입력값 유효성 테스트는 사용자가 입력한 값이 서비스의 규칙에 맞는지 확인하는 테스트다.

회원가입에서는 아래 값들이 대표적인 확인 대상이 된다.

| 입력 항목 | 확인할 것 |
| --- | --- |
| 이메일 | 형식, 중복 여부 |
| 비밀번호 | 길이, 문자 조합, 확인 비밀번호 일치 여부 |
| 전화번호 | 숫자 형식, 길이, 인증 여부 |
| 필수 정보 | 누락 여부 |

입력값 테스트는 케이스가 많아지기 쉽기 때문에, 동등 분할과 경계값 분석을 사용하면 효율적으로 줄일 수 있다.

## 동등 분할

동등 분할은 입력 데이터를 비슷한 특성을 가진 그룹으로 나누고, 각 그룹에서 대표값을 선택해 테스트하는 기법이다.

예를 들어 비밀번호 규칙이 `8자 이상 16자 이하`라면 아래처럼 나눌 수 있다.

| 그룹 | 의미 | 대표값 |
| --- | --- | --- |
| 8자 미만 | 유효하지 않은 값 | 4자 |
| 8자 이상 16자 이하 | 유효한 값 | 12자 |
| 16자 초과 | 유효하지 않은 값 | 20자 |

모든 길이를 다 테스트하지 않아도, 각 그룹의 대표값을 확인하면 입력 규칙이 제대로 적용되는지 효율적으로 볼 수 있다.

## 경계값 분석

경계값 분석은 오류가 자주 발생하는 범위의 경계와 그 주변 값을 집중적으로 확인하는 기법이다.

비밀번호 규칙이 `8자 이상 16자 이하`라면 아래 값이 중요하다.

| 값 | 기대 결과 |
| --- | --- |
| 7자 | 실패 |
| 8자 | 성공 |
| 16자 | 성공 |
| 17자 | 실패 |

::: info 실무 포인트
입력 제한은 경계에서 오류가 많이 발생한다. `8자 이상` 조건이라면 8자만 볼 것이 아니라 7자도 함께 확인해야 한다.
:::

## 3. 예외 상황 및 오류 처리 테스트

예외 상황 테스트는 기능이 실패했을 때 서비스가 안전하고 친절하게 반응하는지 확인하는 과정이다.

| 예외 상황 | 기대 메시지 또는 처리 |
| --- | --- |
| 이미 가입된 이메일 | 이미 사용 중인 이메일입니다. |
| 네트워크 연결 끊김 | 네트워크 연결을 확인해주세요. |
| 비밀번호 형식 오류 | 비밀번호는 8자 이상, 영문/숫자/특수문자를 포함해야 합니다. |
| 서버 오류 | 일시적인 서버 오류입니다. 잠시 후 다시 시도해주세요. |
| 인증번호 불일치 | 인증번호가 일치하지 않습니다. |
| 필수 정보 누락 | 필수 정보를 모두 입력해주세요. |

예외 상황에서는 단순히 에러가 발생하는지만 보면 부족하다. 사용자에게 명확한 안내를 제공하고, 시스템의 데이터 무결성이 유지되는지까지 확인해야 한다.

## 테스트 케이스 구조 예시

잘 구조화된 테스트 케이스는 테스트의 명확성을 높이고 팀원 간 의사소통을 쉽게 만든다.

| 항목 | 예시 |
| --- | --- |
| 테스트 ID | SIGNUP-VALID-001 |
| 분류 | 입력값 유효성 |
| 테스트 목적 | 비밀번호 8자 미만 입력 시 가입이 차단되는지 확인 |
| 사전 조건 | 회원가입 페이지에 접근한 상태 |
| 테스트 데이터 | 비밀번호 `abc12` |
| 테스트 절차 | 필수 정보를 입력하고 5자 비밀번호로 가입 시도 |
| 기대 결과 | 비밀번호 규칙 안내 메시지가 표시되고 가입되지 않음 |
| 실제 결과 | 테스트 실행 후 기록 |
| 판정 | Pass 또는 Fail |

## 분할 기준별 케이스 예시

회원가입 기능을 세 가지 기준으로 나누면 아래처럼 테스트 케이스 묶음을 만들 수 있다.

| 분류 | 케이스 예시 |
| --- | --- |
| 정상 흐름 | 유효한 이메일, 비밀번호, 약관 동의 후 가입 성공 |
| 입력값 유효성 | 이메일 형식 오류, 비밀번호 길이 오류, 필수값 누락 |
| 예외 상황 | 중복 이메일, 서버 오류, 네트워크 끊김, 인증번호 불일치 |

이렇게 나누면 복잡한 기능도 빠뜨리지 않고 체계적으로 테스트할 수 있다.
