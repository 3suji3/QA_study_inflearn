# 실습 결과물로 면접 답변 구성하기

::: tip 핵심 요약
면접 답변은 추상적인 장점보다 직접 정리한 테스트 케이스, 체크리스트, 이슈 기록 같은 결과물을 근거로 구성해야 설득력이 생긴다.
:::

## 왜 실습 결과물이 중요한가?

면접에서 “꼼꼼합니다”라고 말하는 것만으로는 QA 역량을 충분히 보여주기 어렵다.

면접관은 지원자가 실제로 기능을 어떻게 분석하고, 어떤 기준으로 테스트 범위를 정하고, 어떤 방식으로 문제를 발견하고 공유하는지 알고 싶어 한다.

| 추상적인 답변 | 실습 기반 답변 |
| --- | --- |
| 꼼꼼하게 테스트했습니다. | 회원가입 기능을 정상 흐름, 입력값 유효성, 예외 상황으로 나누어 테스트했습니다. |
| 버그를 잘 찾겠습니다. | 오류 메시지, 상태 변화, 데이터 연동을 관찰해 사용자 영향이 있는 문제를 기록했습니다. |
| 열심히 배우겠습니다. | 테스트 케이스와 체크리스트를 직접 작성하며 QA 사고 흐름을 정리했습니다. |

실습 결과물은 “제가 이렇게 생각하고 테스트할 수 있습니다”를 보여주는 증거가 된다.

## 질문 1. 테스트 설계는 어떻게 진행하셨나요?

이 질문은 테스트 설계 과정을 논리적으로 설명할 수 있는지 확인하는 질문이다.

### 답변 구조

| 단계 | 설명 | 답변에 넣을 내용 |
| --- | --- | --- |
| 기능 분석 | 요구사항과 기능 목적을 이해 | 어떤 기능을 테스트했는지 |
| 범위 설정 | 사용자 영향도와 리스크 기준으로 우선순위 결정 | 왜 이 범위를 먼저 봤는지 |
| 케이스 분류 | 정상 흐름, 입력값 유효성, 예외 상황으로 분리 | 어떤 유형으로 나누었는지 |
| 항목 정의 | 목적, 사전 조건, 절차, 기대 결과 작성 | 테스트 케이스를 어떻게 문서화했는지 |

### 답변 예시

회원가입 기능을 기준으로 먼저 요구사항과 사용자 흐름을 확인했습니다. 이후 사용자 영향도가 큰 정상 가입 흐름을 우선순위로 두고, 입력값 유효성, 중복 이메일, 네트워크 오류 같은 예외 상황으로 케이스를 분류했습니다. 각 케이스에는 테스트 목적, 사전 조건, 테스트 절차, 기대 결과를 명확히 작성해 실행 결과를 객관적으로 판단할 수 있게 했습니다.

## 질문 2. 문제를 발견했을 때 어떻게 해결했나요?

이 질문은 단순히 버그를 찾는 능력보다, 문제를 관찰하고 기록하고 공유하는 과정을 확인하는 질문이다.

### QA 스토리보드

| 단계 | 설명 | 답변 포인트 |
| --- | --- | --- |
| 관찰 | 화면 변화, 오류 메시지, 데이터 상태를 세심하게 봄 | 어떤 이상 징후를 봤는지 |
| 발견 | 기대 결과와 실제 결과의 차이를 찾음 | 무엇이 문제였는지 |
| 보고 | 재현 절차, 기대 결과, 실제 결과를 정리 | 개발팀이 재현 가능하게 전달했는지 |
| 배운 점 | 협업 방식이나 테스트 관점 개선 | 다음 테스트에 어떻게 반영했는지 |

### 답변 예시

회원가입 테스트 중 비밀번호 규칙에 맞지 않는 값을 입력했는데도 명확한 오류 메시지가 표시되지 않는 상황을 발견했습니다. 기대 결과는 비밀번호 규칙 안내 메시지가 표시되고 가입 처리가 차단되는 것이었지만, 실제로는 사용자가 무엇을 잘못했는지 알기 어려웠습니다. 해당 내용을 재현 절차와 기대 결과, 실제 결과로 정리해 공유했고, 이후 오류 메시지는 사용자에게 다음 행동을 안내할 수 있어야 한다는 관점을 더 신경 쓰게 되었습니다.

## 질문 3. QA로서 어떤 판단 기준을 가지고 있나요?

이 질문은 지원자가 단순 테스터가 아니라 품질을 판단하는 사람으로 생각하고 있는지 확인하는 질문이다.

| 판단 기준 | 설명 |
| --- | --- |
| 사용자 영향도 | 문제가 실제 사용자에게 얼마나 큰 불편을 주는지 |
| 실패 리스크 | 금전 손실, 데이터 불일치, 개인정보 노출 가능성이 있는지 |
| 재현 가능성 | 같은 조건에서 문제가 반복되는지 |
| 수정 우선순위 | 배포 전 반드시 해결해야 하는 문제인지 |
| 남은 리스크 | 테스트하지 못한 범위나 확인이 필요한 부분이 있는지 |

### 답변 예시

QA는 단순히 버그를 찾는 역할을 넘어, 이 기능이 사용자에게 나가도 되는지 판단해야 한다고 생각합니다. 그래서 이슈를 볼 때 단순 발생 여부만 보지 않고 사용자 영향도, 실패 시 리스크, 재현 가능성을 함께 확인합니다. 예를 들어 결제나 회원가입처럼 사용자 영향이 큰 기능은 오류 메시지뿐 아니라 데이터 생성 여부와 중복 요청 가능성까지 확인해 배포 전 리스크를 줄이려고 합니다.

## 면접 스크립트 점검 방법

답변은 작성한 뒤 실제로 말해보며 점검해야 한다.

| 점검 항목 | 확인할 것 |
| --- | --- |
| 흐름 | 질문에 대한 답이 기능 분석 -> 설계 -> 결과로 이어지는지 |
| 구체성 | 기능명, 테스트 기준, 예시가 들어갔는지 |
| 길이 | 너무 길어서 핵심이 흐려지지 않는지 |
| 판단 기준 | 단순 수행이 아니라 QA의 판단이 드러나는지 |
| 말하기 | 소리 내어 읽었을 때 자연스럽게 들리는지 |

녹음해서 들어보면 문장이 너무 길거나 반복되는 부분을 쉽게 찾을 수 있다.

## 바로 쓸 수 있는 답변 공식

```text
저는 [기능]을 테스트할 때 [사용자 영향도/리스크]를 기준으로 우선순위를 정했습니다.
먼저 [정상 흐름]을 확인하고, 이후 [입력값/상태 변화/예외 상황]으로 케이스를 나누었습니다.
문제가 발견되면 [재현 절차/기대 결과/실제 결과]를 정리해 공유했고,
이를 통해 [배운 점/개선한 점]을 얻었습니다.
```

## Bad vs Good 답변

| 구분 | 답변 | 판단 |
| --- | --- | --- |
| Bad | 꼼꼼하게 테스트했습니다. | 구체적인 기준과 사례가 없음 |
| Bad | 경계값 분석을 사용했습니다. | 용어만 있고 적용 대상이 없음 |
| Good | 비밀번호 규칙 테스트에서 7자, 8자, 16자, 17자 경계값을 확인했습니다. | 기법을 실제 케이스에 적용함 |
| Good | 중복 이메일 입력 시 오류 메시지와 가입 차단 여부를 함께 확인했습니다. | 사용자 안내와 시스템 처리까지 봄 |

## 면접 질문 예시

> 실습한 테스트 결과물을 바탕으로 본인의 QA 역량을 설명해보세요.

## 면접 답변 예시

회원가입 기능을 기준으로 테스트 케이스와 체크리스트를 직접 구성했습니다. 먼저 요구사항과 사용자 흐름을 분석하고, 사용자 영향도가 큰 정상 가입 흐름부터 우선순위를 정했습니다. 이후 입력값 유효성, 중복 이메일, 네트워크 오류, 새로고침과 뒤로가기 같은 상태 변화로 케이스를 나누었습니다. 테스트 중 발견한 문제는 재현 절차, 기대 결과, 실제 결과로 정리해 공유하는 방식으로 기록했고, 이를 통해 QA는 단순히 버그를 찾는 것이 아니라 사용자 영향도와 리스크를 기준으로 배포 가능 여부를 판단하는 역할이라는 점을 배웠습니다.

## 복습 질문

- 면접에서 “꼼꼼합니다”라는 답변이 부족하게 들리는 이유는 무엇일까?
- 테스트 설계 답변에 기능 분석, 범위 설정, 케이스 분류가 들어가야 하는 이유는 무엇일까?
- 문제 해결 경험을 말할 때 관찰, 발견, 보고, 배운 점 순서가 왜 좋을까?
- 내가 정리한 테스트 케이스 중 면접 답변에 활용하기 좋은 사례는 무엇일까?
