# 실무 QA 보너스 지식 지도

::: tip 핵심 요약
강의 노트가 테스트 설계와 면접 준비의 뼈대라면, 보너스 지식은 실제 현장에서 자주 마주치는 업무 맥락을 채워주는 보조 지도다.
:::

## 왜 보너스 지식이 필요할까?

테스트 케이스를 잘 쓰는 것만으로 실무 QA가 끝나지는 않는다. 실제 업무에서는 테스트 환경이 다르거나, 데이터가 꼬이거나, 이슈 우선순위를 정해야 하거나, 배포 가능 여부를 판단해야 하는 상황이 자주 생긴다.

이 섹션은 그런 상황에서 덜 당황하도록 최소한 알아두면 좋은 내용을 정리한다.

## 먼저 볼 순서

| 순서 | 문서 | 왜 필요한가 |
| --- | --- | --- |
| 1 | [테스트 종류와 전략](/bonus/test-types-and-strategy) | 지금 하는 테스트가 어떤 성격인지 구분하기 위해 |
| 2 | [결함 관리와 릴리즈 판단](/bonus/defect-and-release) | 버그를 어떻게 분류하고 배포 여부를 판단할지 알기 위해 |
| 3 | [테스트 환경과 데이터 관리](/bonus/test-environment-and-data) | 환경 문제와 기능 문제를 구분하고 안정적으로 테스트하기 위해 |

## 실무 QA가 자주 하는 판단

| 상황 | QA가 판단할 것 |
| --- | --- |
| 시간이 부족함 | 어떤 기능을 먼저 테스트할지 |
| 버그가 발견됨 | 심각도와 우선순위가 어느 정도인지 |
| 수정 완료됨 | 재테스트만 할지, 회귀 테스트까지 할지 |
| 배포가 임박함 | 남은 리스크를 수용할 수 있는지 |
| 테스트가 막힘 | 환경 문제인지, 기능 문제인지, 데이터 문제인지 |

::: info 실무 포인트
QA는 모든 문제를 혼자 해결하는 사람이 아니라, 팀이 올바른 품질 판단을 할 수 있도록 근거를 정리하는 사람이다.
:::

## 강의 노트와 연결하기

| 강의 노트 | 연결되는 보너스 지식 |
| --- | --- |
| 테스트 범위 설정 | 테스트 종류, 우선순위, 리스크 |
| 테스트 케이스 작성 | 테스트 데이터, 사전 조건, 기대 결과 |
| 관찰 포인트 | 화면 상태, 데이터 연동, 로그 |
| 체크리스트 적용 | 환경 준비, 결과 공유, 재테스트 |
| 면접 답변 구성 | 결함 분류, 배포 판단, QA 역할 설명 |

## 복습 질문

- 내가 발견한 이슈가 배포를 막을 정도인지 판단하려면 어떤 정보가 필요할까?
- 테스트가 실패했을 때 환경 문제와 기능 문제를 어떻게 구분할 수 있을까?
- 재테스트와 회귀 테스트는 언제 각각 필요할까?
