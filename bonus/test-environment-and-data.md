# 테스트 환경과 데이터 관리

::: tip 핵심 요약
테스트가 실패했을 때 기능 문제인지, 환경 문제인지, 데이터 문제인지 구분하려면 테스트 환경과 데이터를 의식적으로 관리해야 한다.
:::

## 테스트 환경이란?

테스트 환경은 기능을 검증하는 데 필요한 실행 조건 전체를 말한다.

| 항목 | 예시 |
| --- | --- |
| 서버 환경 | Local, Dev, QA, Staging, Production |
| 클라이언트 환경 | Windows, macOS, Android, iOS |
| 브라우저 | Chrome, Edge, Safari |
| 앱 버전 | v1.0.0, v1.1.0 |
| 계정 권한 | 일반 사용자, 관리자, 제한 계정 |
| 네트워크 | Wi-Fi, 5G, 오프라인 |

같은 기능도 환경이 다르면 결과가 달라질 수 있다. 그래서 버그 리포트에는 환경 정보를 반드시 남겨야 한다.

## 환경별 특징

| 환경 | 의미 | 주의할 점 |
| --- | --- | --- |
| Local | 개발자 개인 환경 | 실제 배포 환경과 다를 수 있음 |
| Dev | 개발 통합 환경 | 기능이 자주 바뀜 |
| QA | QA 검증 환경 | 테스트 데이터와 설정 확인 필요 |
| Staging | 운영과 유사한 사전 배포 환경 | 최종 검증에 사용 |
| Production | 실제 운영 환경 | 테스트 행동이 실제 사용자에게 영향 가능 |

운영 환경에서 테스트할 때는 실제 데이터 생성, 결제, 알림 발송 같은 부작용을 특히 조심해야 한다.

## 테스트 데이터 관리

테스트 데이터는 테스트 결과를 좌우한다.

| 데이터 유형 | 예시 | 필요한 이유 |
| --- | --- | --- |
| 정상 데이터 | 신규 이메일, 유효한 비밀번호 | 정상 흐름 확인 |
| 예외 데이터 | 잘못된 이메일, 짧은 비밀번호 | 유효성 검사 확인 |
| 중복 데이터 | 이미 가입된 이메일 | 중복 처리 확인 |
| 경계 데이터 | 7자, 8자, 16자, 17자 | 경계값 분석 |
| 권한 데이터 | 관리자, 일반 사용자 | 접근 제어 확인 |

## 테스트 전 데이터 체크

1. 사용할 계정이 어떤 상태인지 확인한다.
2. 이미 가입된 이메일인지, 신규 이메일인지 구분한다.
3. 테스트 후 데이터가 남아도 되는지 확인한다.
4. 결제나 알림처럼 실제 비용이 발생하는지 확인한다.
5. 여러 테스트가 같은 데이터를 공유해도 되는지 확인한다.

## 로그와 증거 자료

문제가 발생했을 때는 화면만 보는 것보다 증거를 함께 남기는 것이 좋다.

| 자료 | 왜 필요한가 |
| --- | --- |
| 스크린샷 | 화면 상태와 메시지 확인 |
| 화면 녹화 | 재현 흐름 확인 |
| 콘솔 로그 | 브라우저 오류 확인 |
| 네트워크 로그 | API 요청과 응답 확인 |
| 서버 로그 | 백엔드 처리 오류 확인 |

초보 QA라도 브라우저 개발자 도구의 `Console`과 `Network` 탭 이름 정도는 익혀두면 협업에 도움이 된다.

## 환경 문제와 기능 문제 구분하기

| 증상 | 먼저 확인할 것 |
| --- | --- |
| 나만 안 됨 | 브라우저, 캐시, 계정 상태 |
| 모두 안 됨 | 서버 상태, 배포 버전 |
| 특정 데이터만 안 됨 | 테스트 데이터 상태 |
| API 응답이 이상함 | Network 탭, 서버 로그 |
| 화면만 깨짐 | 브라우저, 해상도, CSS 영향 |

## 복습 질문

- 버그 리포트에 환경 정보를 남겨야 하는 이유는 무엇일까?
- 신규 이메일과 중복 이메일은 왜 테스트 데이터로 구분해야 할까?
- 문제가 발생했을 때 스크린샷 외에 어떤 증거 자료를 남길 수 있을까?
