# 테스트 종류와 전략

::: tip 핵심 요약
테스트 종류를 구분할 수 있으면 “무엇을 얼마나 깊게 확인해야 하는지”를 더 정확하게 판단할 수 있다.
:::

## 테스트 종류를 알아야 하는 이유

실무에서는 “테스트해 주세요”라는 요청만으로는 범위가 충분히 명확하지 않다. QA는 지금 필요한 테스트가 기본 동작 확인인지, 수정 사항 재검증인지, 기존 기능 영향 확인인지 구분해야 한다.

| 테스트 종류 | 목적 | 언제 쓰는가 |
| --- | --- | --- |
| Smoke Test | 빌드가 테스트 가능한 상태인지 빠르게 확인 | 새 빌드 배포 직후 |
| Sanity Test | 특정 수정이 정상인지 좁게 확인 | 작은 수정 후 |
| Regression Test | 기존 기능이 깨지지 않았는지 확인 | 수정/배포 전 |
| Functional Test | 기능이 요구사항대로 동작하는지 확인 | 기능 검증 시 |
| Exploratory Testing | 정해진 케이스 밖에서 의심하며 탐색 | 복잡한 기능이나 신규 기능 확인 시 |
| UAT | 사용자/현업 관점에서 완료 여부 확인 | 출시 전 최종 확인 |

## 수동 테스트와 자동화 테스트

| 구분 | 수동 테스트 | 자동화 테스트 |
| --- | --- | --- |
| 강점 | 사용자 관점, 탐색, UX 확인에 강함 | 반복 검증, 회귀 테스트에 강함 |
| 약점 | 시간이 오래 걸리고 반복에 취약 | 초기 구축 비용과 유지보수 필요 |
| 예시 | 회원가입 흐름 직접 확인 | 로그인 성공/실패 케이스 자동 실행 |

자동화는 수동 테스트를 대체하는 것이 아니라 반복되는 검증을 줄여주는 도구다. 신규 기능의 UX, 오류 메시지, 애매한 상태 변화는 여전히 사람이 관찰해야 하는 경우가 많다.

## 기능 테스트와 비기능 테스트

| 구분 | 확인하는 것 | 예시 |
| --- | --- | --- |
| 기능 테스트 | 요구사항대로 동작하는가 | 회원가입 성공, 로그인 실패 메시지 |
| 비기능 테스트 | 얼마나 안정적이고 사용하기 좋은가 | 성능, 보안, 접근성, 호환성 |

초보 QA는 기능 테스트부터 시작해도 좋지만, 실무에서는 비기능 관점도 조금씩 챙겨야 한다.

## 테스트 전략을 세우는 순서

1. 변경된 기능을 확인한다.
2. 사용자 영향이 큰 흐름을 찾는다.
3. 정상 흐름을 먼저 확인한다.
4. 입력값과 예외 상황을 나눈다.
5. 수정 영향이 있는 주변 기능을 회귀 테스트 범위에 넣는다.
6. 남은 리스크를 팀에 공유한다.

## 예시: 회원가입 기능

| 테스트 목적 | 선택할 테스트 |
| --- | --- |
| 새 배포가 기본적으로 동작하는지 확인 | Smoke Test |
| 회원가입 버튼 오류 수정 확인 | Sanity Test 또는 Re-test |
| 회원가입 수정 후 로그인도 정상인지 확인 | Regression Test |
| 이메일/비밀번호 규칙 검증 | Functional Test |
| 예상 밖 입력과 화면 흐름 탐색 | Exploratory Testing |

## 복습 질문

- Smoke Test와 Regression Test는 어떤 상황에서 다르게 쓰일까?
- 자동화 테스트가 있어도 수동 테스트가 필요한 이유는 무엇일까?
- 회원가입 기능을 수정했을 때 로그인까지 확인해야 할 수 있는 이유는 무엇일까?
