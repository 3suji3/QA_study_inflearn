# 결함 관리와 릴리즈 판단

::: tip 핵심 요약
버그를 찾는 것보다 중요한 일은 결함의 영향도를 판단하고, 팀이 배포 여부를 결정할 수 있게 근거를 정리하는 것이다.
:::

## 결함 생명주기

실무에서 버그는 발견하고 끝나는 것이 아니라 상태를 따라 관리된다.

| 상태 | 의미 |
| --- | --- |
| Open | 이슈가 새로 등록됨 |
| In Progress | 개발자가 수정 중 |
| Fixed | 수정 완료 |
| Ready for QA | QA가 재테스트할 수 있는 상태 |
| Reopened | 수정 후에도 문제가 남아 다시 열림 |
| Closed | QA 확인 후 종료 |
| Deferred | 이번 배포에서는 처리하지 않고 연기 |

QA는 각 상태에서 무엇을 확인해야 하는지 알아야 한다. 특히 `Fixed`라고 해서 바로 끝난 것이 아니라, QA가 재테스트해서 실제 해결 여부를 확인해야 한다.

## Severity와 Priority

| 구분 | 의미 | 예시 |
| --- | --- | --- |
| Severity | 문제가 서비스에 주는 영향 | 결제 불가, 앱 종료, 개인정보 노출 |
| Priority | 얼마나 빨리 처리해야 하는지 | 오늘 배포 전 수정, 다음 스프린트 처리 |

심각도는 “얼마나 큰 문제인가”이고, 우선순위는 “얼마나 빨리 처리할 것인가”다.

## 결함을 분류하는 기준

| 등급 | 판단 기준 | 예시 |
| --- | --- | --- |
| Critical | 핵심 기능 불가, 금전/데이터 손실 위험 | 결제 완료 후 주문 미생성 |
| Major | 주요 기능에 큰 불편 발생 | 로그인 실패 메시지 미노출 |
| Minor | 우회 가능하지만 수정 필요 | 특정 문구 오탈자 |
| Trivial | 사용 영향이 매우 작음 | 여백, 정렬 미세 차이 |

팀마다 등급 이름과 기준은 다를 수 있다. 중요한 것은 이슈를 등록할 때 사용자 영향과 재현 조건을 함께 적는 것이다.

## 릴리즈 판단 기준

배포 전 QA는 아래 질문에 답할 수 있어야 한다.

| 질문 | 확인할 것 |
| --- | --- |
| 핵심 기능이 정상인가? | 회원가입, 로그인, 결제 같은 주요 흐름 |
| 치명적인 결함이 남아 있는가? | 배포를 막을 정도의 이슈 |
| 수정된 이슈는 재테스트했는가? | Fixed 이슈의 실제 해결 여부 |
| 기존 기능은 깨지지 않았는가? | 회귀 테스트 결과 |
| 남은 리스크는 공유되었는가? | 제외 범위, 미확인 항목, 알려진 이슈 |

::: warning 주의
QA 승인은 “버그가 하나도 없다”는 뜻이 아니다. 확인한 범위와 남은 리스크를 기준으로 배포 가능한 수준인지 판단하는 것이다.
:::

## Go / No-Go 판단

| 판단 | 의미 | 예시 |
| --- | --- | --- |
| Go | 배포 가능 | 치명적 이슈 없음, 핵심 흐름 통과 |
| Conditional Go | 조건부 배포 가능 | 경미한 이슈는 남았지만 영향과 우회 방법 공유됨 |
| No-Go | 배포 불가 | 결제 불가, 데이터 손상 등 치명적 이슈 존재 |

## 복습 질문

- `Fixed` 상태의 이슈를 QA가 바로 닫으면 안 되는 이유는 무엇일까?
- Severity와 Priority를 구분해야 하는 이유는 무엇일까?
- QA 승인이 “버그 없음”을 의미하지 않는 이유는 무엇일까?
